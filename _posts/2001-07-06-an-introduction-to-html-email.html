So, you want to send HTML email to your clients, prospects\nor newsletter subscribers? Marketing has descended from upon high\nand declared it, the small business client wants it, an executive\nin management has read about it.

Well, why not? The click-through rates are noticeably\nhigher on HTML email. Analysis shows that customers are less likely\nto unsubscribe from HTML email than their text counterparts. In\nmy last TemplateKit\nemail newsletter, I had 11 unsubscribes, 10 of which were text\nrecipients. Customers simply respond with more click-throughs, more\nsales, more inquiries for information, if your message is in a form\nthat the recipient can easily view and display correctly.

The fact of the matter is that email HTML browsers are\njust not equal to their web browser equivalents. This is further\ncomplicated by the wide variety of settings, preferences, security\nupdates, versions, and third-party applications which make the user\nexperience hard to predict.

Cutting and Pasting from a Web page

There are several ways to get HTML code from an existing document\nwithout using an HTML editor.

You can "view source" and copy the code to the clipboard;\ncreate a copy with the "save as" command; or you can try\nto select the code from the body of the document and paste it into\nanother program.

Notice how links inside that document are set up. If your document\ncontains a number of relative links, (i.e. /images/yourimage.gif),\nthen you might not be able to find your images when it is time to\npreview and test.

You may want to include a <BASE> tag in your HTML email so\nyour recipient's browser knows how to interpret document relative\nlinks.

The BASE element allows global reference information to be set.\nUse of the HREF attribute provides a base address for interpreting\nall relative URL references in a document when the document is read\nout of context. The TARGET attribute specifies a global frame destination\nname to be used for page activation changes (in links, forms and\nimage maps.)

Example:

<BASE HREF="http://www.florentinedesign.com" TARGET="_top"> \n

This may help your mailings, however we would recommend an absolute\nURL for each of your links.

Cutting and Pasting from a Microsoft Office Product

If you cut and paste from Microsoft Word, there is always the possibility\nthat some formatting will not carry over into HTML very well. This\nis most often seen in the case of Auto formatting, when MS Word\nconverts common keystrokes into symbols. -, "", ©, and a host of others. These are called Windows Characters, and\nthey are not interpreted by your web browser.

Word represents these ASCII characters as numeric values which a\nbrowser cannot understand.

This is why it is best to always work in text mode, or save your\ndocument as a dos text document and lose all formatting before transferring\nit to your HTML email.

An excellent Text Editor to use is TextPad,\nwhich allows you to specify which code set you should use and\nsave your document as. Be aware that merely pasting your code into\nTextPad or\nsaving your document in MS Word as a text document is not good enough.

Using a Previously Mailed Document as the Basis for your code.

You may experience problems whenever you try to use code, which\nhas already been mailed, as the basis for your new mailing. This\nis most often seen with code that has been rendered through a web-based\nbrowser which you wish to use again. Many web-based email clients\nand image hosting companies, such as Netscape NetMail or Akamai,\nhave their own redirection and accounting which break your links\nif you try to send them a link which has already been processed\nby their system before.

Simply put, do not try to redirect a redirect. This is true for\nyour own redirects as well.

Since these systems assign these redirects to any HTML link that\nit can discover, it is often better to avoid using "named anchors"\nas well. A named anchor, or "jump anchor" is a link which\ntakes you to another spot in the same document.

<A Name="foo">

If a redirect gets placed on a named anchor, it will most likely\neither break or spawn a new browser window with unpredictable results.\nFrom a marketing standpoint, if your email is so long that you feel\nyou need to use named anchors, maybe you should consider editing.
