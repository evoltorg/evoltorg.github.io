Headings as they stand today

The headings <h1> through <h6> \nare very basic ways of representing headings within a document. \nAlthough the method stands up well enough to mark up basic \ndocuments, it doesn't stand up to everything we can throw at it.

It would have been much nicer if from the start the header tags had \nbeen defined something along the lines of the following, from day one:

\n<h1 text="top heading">\n	<h2 text="lower heading">\n		<p>random content</p>\n	</h2>\n	<p>text unassociated with the h2</p>\n</h1>\n

A refinement of headings

Obviously though, this is not the case. This doesn't mean that \nwe're not able to associate headers with their content, however, just that \nwe have to do it ourselves. This is where the extensibility of XML \ncan help you if you're using XHTML.

Because of the namespacing abilities of XML we can mix in our \nown custom markup with our XHTML. We can use this to produce a \nfully degradable (HTML browsers are supposed to ignore unknown \nelements) container element to associate our headings with their \ncontent.

First of all we need to declare the namespace for our heading \nextensions. For the purposes of this exercise let's use the URL \nhttp://members.evolt.org/luminosity/xml/header/. Now, to \nincorporate our extended elements we change our base tag from \n<html xmlns="http://www.w3.org/1999/xhtml" xml:lang="en">, \nto <html xmlns="http://www.w3.org/1999/xhtml" \nxmlns:header="http://members.evolt.org/luminosity/xml/header/" xml:lang="en">.

Now whenever we run across a header we can associate it with \nits content using our custom designed tags. Every time you see a header, \nput a <header:hn> before the header, and a \n</header:hn> after the last of its associated \ncontent. For example, if we start off with the following code sample:

\n<html xmlns="http://www.w3.org/1999/xhtml" xml:lang="en">\n	<head>\n		<title>The supermarket fiasco</title>\n	</head>\n	<body>\n		<h1>The supermarket fiasco</h1>\n		<h2>The beginning</h2>\n		<p>In the beginning was the supermarket.\n		The supermarket dished out your typical\n		bland supermarket food.</p>\n		…\n		<h2>The end</h2>\n		<p>And so the public cheered, as the last\n		supermarket burnt, and their taste buds\n		were once again put to use.</p>\n	</body>\n</html>\n

We replace it with this:

\n<html xmlns="http://www.w3.org/1999/xhtml" xmlns:header="http://members.evolt.org/luminosity/xml/header/" xml:lang="en">\n	<head>\n		<title>The supermarket fiasco</title>\n	</head>\n	<body>\n		<header:h1>\n		<h1>The supermarket fiasco</h1>\n		<header:h2>\n		<h2>The beginning</h2>\n		<p>In the beginning was the supermarket.\n		The supermarket dished out your typical\n		bland supermarket food.</p>\n		…\n		</header:h2>\n		<header:h2>\n		<h2>The end</h2>\n		<p>And so the public cheered, as the last\n		supermarket burnt, and their taste buds\n		were once again put to use.</p>\n		</header:h2>\n		</header:h1>\n	</body>\n</html>\n

The optional extras

So how does this help us? Well it satisifies those \nof us who wish that header had been a container element \nfor its content all along. You can regard the h1 inside \nour h1 the same as the legend inside a fieldset tag, if \nyou're not happy with having the same data seemingly \ndefined twice. You can also set the text of the original \nh1 to be an attribute for your new h1 if you want. Use \nsomething like: <header:h1 header:text="The supermarket fiasco">. \nIn the future, when browser support is good enough you \ncould then eliminate the original hn tags and use the \nbefore pseduo-class with generated content to print out \nthe header (header:hn:before {content: attr(header:text);}), \nalthough I would recommend against this, because your \nnew element won't have semantic meaning to browsers, \nwhereas the old hn tags do.

We can of course use the new tags to style all content \nbased on their level of heading — in fact this would be \nthe only practical usage that I can imagine at this time. \nSomething along the lines of:

\nheader:h1 {\n	background-color: blue;\n}
header:h2 {\n	background-color: light blue;\n}\n

And of course, you can further extend the new headers \nwith your own attributes, and you could then use them with \nsome nifty JavaScript or what have you.

Also, you're not limited to extending header elements. \nYou can also use your own namespaces to create documents that use \nyour own tags to fill in for other elements that you need, or want, \nthat HTML doesn't provide yet.
