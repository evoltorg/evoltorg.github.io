\n A common requirement of sites with registered users or members is to be able\n to email different groups of members with information that might be of interest\n or use to them.\n

\n

\n This might be with notifications of new content, matching users'\n preferences and interests, or it could be more generic send an email to\n all of such-and-such a group business requirement. In either case,\n you'll probably be interested in respecting the \n permissions\n your users have given you to contact them.\n

\n

\n This will always mean only sending emails to people who have given you\n permission to do so, and often matching the emails to people whose interest\n you know matches the thing you want to send.\n

\n

\n Because you're running your site professionally (you are, right?), you'll\n also want to send emails as part of a workflow. This means that only appropriate\n people will have the rights to send email, and you'll have an audit trail\n of who sent what, when. That way, if you get complaints of spam, you can\n quickly find out the facts (You gave us permission to do it and have\n an interest in the subject, or otherwise) and respond appropriately.\n

\n

Business Requirements

\n

\n For a system which announces new content on a site, our system \n requirements are basically:\n

\n
    \n
  1. \n To be able to send email to registered users of the site.\n
    2. \n
  2. \n To enable users to give and retract permission for you to email them,\n and for your email dispatch to respect that permission.\n
    3. \n
  3. \n To enable users to register areas of interest, and for your email dispatch\n to only send email to users who are interested in areas which relate to\n what you propose to send.\n
    4. \n
  4. \n To have emails only sent out when the content is signed off for dispatch.\n
    5. \n
  5. \n To have the email contain \n
      \n
    1. A standard short message
      2. \n
    2. A paragraph of content-specific text \n (this will be a synopsis of the content, and is an existing data \n field in our content schema)
      3. \n
    3. A link to the content
      4. \n
    \n
    6. \n
  6. \n To have the person who sends the announcement (the actor in the\n workflow transition, in UML-speak) receive a notification of who the\n email announcement went to.\n
    7. \n \n

\n We could have a requirement that emails are sent on a schedule,\n and pick up all new content published in the last n hours.\n But for simplicity, that's not how I've chosen to do it. The normal thing\n to say here is that that's left as an exercise to the\n reader.\n

System

\n

\n The system I'll use to demonstrate this is \n Plone -\n a Zope-based CMS\n which provides a rich API that provides hooks for adding user data elements\n and state & transition workflow scripting, making all of the following \n extremely simple.\n

\n

\n I'm not going to cover installing Zope, Plone, or Plone sites here - there's\n some reasonable install documentation on the Plone site, \n with some handy all-in-one installers for Windows & Mac OSX.\n

\n

\n This should also work on basic CMF, but is untested on\n that platform. Caveat Emptor.\n

Data Requirements

\n

\n The basis of the solution is (pseudo-code):\n

\n

\n If user has emailPermission and\n userInterest(any) matches contentKeyword(any)\n then sendEmail\n \n

\n

\n Therefore, there are four key pieces of data:\n

\n

Content Data

\n
    \n
  1. \n Metadata Keywords
    \n Plone provides this out of the box, with a light\n Dublin Core implementation.\n
    2. \n
\n

User Data

\n
    \n
  1. \n Email Address
    \n Self-evidently, you need this. It's an existing user attribute in\n Plone, and is already required.\n
    2. \n
  2. \n Email Permission
    \n This is a simple user-settable boolean flag to say that the user grants permission\n to you to send them relevant email. Plone does not provide this out of\n the box, so we'll need to add it. We could use the Listed\n user property, but it's less confusing to keep them separate, and gives\n the user better control over their preferences.\n
    3. \n
  3. \n User Interests
    \n This is a list data-type, each element being an area that the user\n is interested in. If it's not there, they're not interested.\n
    4. \n
\n

Implementation

Adding User Data Elements

\n

\n Adding data elements to users in Plone is pretty simple (unlike adding elements\n to content, which is a whole other story). Plone keeps its user data schema in the \n portal_memberdata tool. In the ZMI,\n navigate to there, and select the Properties tab. This will give\n you a list of the currently available user properties, and their default\n values.\n

\n

\n You need to add two new properties (all values is case-sensitive):\n

\n\n \n \n \n \n \n \n \n \n \n \n \n \n \n \n \n
Property NameProperty TypeDefault Value
interestlinesnull
emailPermissionbooleanfalse (ie unchecked)
\n

\n The result is that all current users, and all new users, will have no\n interests registered, and have not granted you permission to email them.\n This is A Good Thing, as your emails will now be opt-in.\n

Enabling Data Entry

\n

\n It's no use having data elements on each user data object if the users can't\n enter data into the waiting slots. So we need to customise the standard\n Plone form that users use to personalise their experience. This form can\n be found at /portal_skins/plone_forms/personalize_form. \n

\n

\n If you're not used to customising CMF/Plone sites, you'll be worried that it's\n not editable. This is because it's looking at your server's file system (which\n Zope won't write to) for the data for this folder. To enable editing, you\n need to transfer the HTML file to the /portal_skins/custom\n folder, where you can edit it. There's a handy button on the locked form\n page, labelled Customize. Push it... you can now edit the form.\n How and why this works is beyond the scope of this article. For now, accept\n that it just does.\n

\n

\n Once you've hit 'Customize', you'll find the HTML in a normal text-area\n form field. Again, don't worry that it appears not to have any of your\n site template in. The CMS is picking the main content out and inserting it into\n a template slot.\n

Enabling the Member Interest Selection

\n

\n We're going to let members select their interests by means of checkboxes. This\n is a bit harder than radio buttons as we don't have existing form code to\n copy, but knowing how to retrieve data values, it's not hard.\n

\n

\n Remember that the selections are going to end up as a list data type? Zope\n is going to help you out in a big way here. Zope has a wonderful shortcut\n to constructing list data - if you label your fields as:\n name="foo:list"\n Zope will auto-magically bundle all the data together and make it available\n as a list called foo. Neat, eh?\n

\n

\n So all we have to do to save the data is make sure that all our checkboxes\n are named interest:list and when we submit, we'll get a list\n saved in the member's interest data element.\n

\n

\n Retrieving the data is also pretty simple. We're using a basic Python list\n function for testing whether a value is a member of a list, and if it\n is, we're writing in the checked attribute. I've only shown\n three checkboxes, but you can have as many as you like in your own layout.\n As long as they're within the <div>, it'll work fine.\n

Enabling the Content Metadata

\n

\n This bit's really easy. Just take keywords you used in the checkboxes above\n and add them to the content, either through the properties\n tab when viewing Plone, or via the portal_metadata tool in\n the ZMI.\n

\n

\n Note that these have to be exactly the field values you\n used for the checkboxes. An easy mistake is to use the field labels.\n

Workflow Scripting

\n

\n Now that all the data's in place, we're on the home straight. All we need\n to do is add a script that compares the content and user data, checks that\n a member has given us permission to email them and fire off a few emails.\n

\n

\n We'll do this in the standard workflow tool, which is the bundled\n DCWorkflow\n product. This is a 'states-and-transitions' type of workflow.\n You set up states that a content object can be in - with permissions attached\n to each state - and define transitions between those states. DCWorkflow \n also lets you apply scripts to execute before and/or after each transition,\n which is what we need.\n

\n

\n Rather than simply adding a script to an existing transition, we're going\n to add a new state and transition specifically for email announcement. This\n will ensure that sending announcements is logged in the standard workflow\n audit trail, so we know whether a piece of content has been announced, and\n if so, when.\n

\n

Adding a Workflow State

\n

\n Go to the portal_workflow tool in the ZMI and select the Contents\n tab. This will give you the workflows that your site is currently using.\n Unless you've done any customisation already, you'll have 2: a folder\n workflow and a Plone workflow. The Plone workflow is the default one which\n controls most normal content, so it's this one we'll be editing.\n

\n

\n Select that workflow and head to the States tab. Add a state called announced.\n This is the destination state that the content will be in after the emails\n have been sent. Set the permissions to a duplicate of the Published\n state. You need to make sure that your email recipients can view the\n content that you're announcing, so you'll want to set View and\n Access Contents Information permissions for\n Anonymous and Authenticated users, and once the content's announced, you\n don't want it being edited without further workflow, so only give the\n Manager role the permission of Modify Portal Content.\n

\n

\n Next, select which transitions Announced content can then undergo. Again,\n I'd duplicate the Published state, and only permit the Reject\n and Retract transitions.\n

Adding a Workflow Transition

\n

\n Back up to the Plone Workflow, and select the Transitions tab and add a\n new transition called 'announce'. The important properties to set are\n that there's a role guard - only the Manager role should be able to\n send email announcements - and in the 'Display in actions' box fields\n have a sensible name (eg Announce by email) and the category 'workflow'.\n Also make sure that the destination state is 'announced' and that the trigger\n is a user action. We'll add a workflow script after the script is set up.\n

\n Enabling the transition\n

\n

\n Site managers will only be able to use the new transition if it's enabled\n as a permitted transition from an existing state. Go to the published state\n and add the 'announce' transition to the possible transitions available from\n the published state.\n

Adding a Workflow Script

\n Workflow scripts can be Python scripts, page templates, DTML documents or\n any other executable content you can add via the ZMI. We're going to use\n a Python script, so go to the workflow's Script tab and add a new script \n called email_announce.\n

\n

\n This will take one parameter, review_state (which refers to\n the transition currently underway). Here's the script. Note that as with \n all Python coding, the indenting is significant.\n

\n
\n#This script has been designed to send email to cmf users with appropriate\n#preferences. The script should be used in conjunction with the workflow tool.  \n#parameters review_state
# Set up a empty list of email addresses\n# loop through the portal membership, pass memberId to check for\n# Member role. If successful, check to see if the member has given\n# permission to send email, and an area of business interest that \n# coincides with a content keyword. If successful, append the \n# list of email addresses and send them email
# Get the content object we're publishing\ncontentObject = review_state.object
# A nifty little function, which checks to see whether there are any elements\n# that match between two lists, and returns the number of matches. Result: if\n# the function returns 'true', you've got a match\ndef isIn(list1, list2):\n     y=0\n     for x in list1:\n         if x in list2:\n             y += 1\n     return y
# Start with an empty list\nmailList=[]
# Iterate through all the site's users\nfor item in context.portal_membership.listMembers():\n    memberId = item.id\n    \n    # Remember that a real name is not mandatory, so fall back to the username\n    if item.fullname:\n        memberName = item.fullname\n    else:\n        memberName = memberId\n    \n    # Get a list of this member's interests...\n    memberInterests = item.interest\n    # ...and another that's the keywords of this object\n    contentKeywords = contentObject.subject\n    \n    # Check to see if there's a match between the two\n    isInterestedIn = isIn(memberInterests, contentKeywords)\n    \n    # This is the key condition:\n    # If the user has the Member role and\n    #             we have an email address and\n    #             the user's interested in this content and\n    #             we have permission to email them\n    if 'Member' in context.portal_membership.getMemberById(memberId).getRoles() and (item.email !='') and isInterestedIn and item.emailPermission:\n        # add them to the list of people we're emailing\n        mailList.append(item.email)\n        # check that we can send email via the Zope standard Mail Host\n        try:\n           mailhost=getattr(context, context.portal_url.superValues('Mail Host')[0].id)\n        except:\n           raise AttributeError, "Cannot find a Mail Host object"\n       \n        # Let's write an email:\n        mMsg = 'Dear ' + memberName + ',
'\n        mMsg += 'We thought you\'d be interested in hearing about:\n'\n        mMsg += contentObject.TitleOrId() + '
'\n        mMsg += 'Description: \n' + contentObject.Description() + '
'\n        mMsg += 'More info at:\n' + contentObject.absolute_url() + '\n'\n        mTo = item.email\n        mFrom = 'you@yoursite.com'\n        mSubj = 'New Content available'
        # and send it\n        mailhost.send(mMsg, mTo, mFrom, mSubj)
# The change in indentation signals the end of the loop, so we've\n# now sent all the emails. Let's now send a confirmation that we've done it.
# We'll be building the email as a string again, but we have to convert our\n# list data elements into a string before we can append the information\nrecipients = string.join(mailList, sep='\n')\nkeywordsString = string.join(contentKeywords, sep='\n')
mTo = 'you@yourdomain.com'\nmMsg = 'The following people were sent a link to\n'\nmMsg += contentObject.absolute_url() + '
'\nmMsg += recipients + '
'\nmMsg += 'The keywords were:\n' + keywordsString\nmSubj = 'Content announcement email confirmation'\nmailhost.send(mMsg, mTo, mFrom, mSubj)\n

\n Once you have the script set up, go back to your announce transition\n and select it in the 'before transition' slot - that way, you'll only\n complete the transition if the email all gets sent.\n

    \n
  1. Added a boolean email permission field to user data
    2. \n
  2. Added a list-type area of interest field to user data
    3. \n
  3. Amended the personalisation form so that users can store their preferences in the new fields
    4. \n
  4. Added appropriate keywords to content
    5. \n
  5. Added a new workflow state and transition to the workflow
    6. \n
  6. Added a script to select suitable members and send them email announcing the new content
    7. \n
