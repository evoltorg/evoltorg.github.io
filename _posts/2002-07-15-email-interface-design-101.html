Note: this article is not about web based interfaces to email, like Yahoo email \n or Hotmail. It's about using email as an interface to websites or web services.

\n

Mailing list managers have long used a command based email interface \n to manage the mailing list. Link based email interfaces (unsubscribe \n links) are common in email newsletters and mailing list, and job websites also \n typically offer email interfaces using links. Online vendors use HTML based \n emails with forms to sell products.

\n

Email as an interface to websites hasn't been explored much by the interaction \n design community. And yet email has some key advantages: it's fast and asynchronous\n (you can fill in the interface, and the next time you connect to the Internet \n send the email), and allows for requested interaction.

Advantages and Disadvantages of Email Interfaces

\n

Email interfaces have some key advantages over web interfaces:

\n
    \n

  1. Asynchronous interaction
    \n You don't have to be online while giving \n the commands. A significant advantage for many people who have limited Internet \n access. [1]

    2. \n

  2. Requested interaction
    Software systems can request interaction by \n sending you an email.

    3. \n

  3. Speed
    \n You can send commands from your email application, which will \n often be faster than opening a website.

    4. \n
\n

A main advantage of email interfaces, asynchronicity, is also its main disadvantage, \n in that there is no immediate feedback. Feedback from the system can \n take over 10 minutes because of email delays on the Internet. This means that \n email interfaces are mainly useful for one step interactions: you get \n some input and you decide on an action. You shouldn't use email interfaces when:

\n
    \n

  1. Immediate feedback to the user is required upon action.
    Email delivery \n can experience delays of 10 minutes or longer.

    2. \n

  2. Tasks require multiple steps in a short time interval.
    \n You don't \n want the person using this interface to have to wait for the next email to \n continue.

    3. \n
\n

The asynchronocity of the interaction also means that UNDO becomes even more \n important, since no immediate feedback is available.

\n

Requested interaction is a unique advantage of email compared to web based \n interfaces, but not limited to requesting action through email interfaces: requests \n can be sent use a web interface as well.

\n

Speed of the email interface is mainly accomplished by not having to leave \n your email program: starting up a browser and connecting to a website can take \n up to 15 seconds, a time delay in which the task might already have been completed \n using an email interface. The time gains are small, but add up for often repeated \n actions.

\n

Input and Output in Email Interfaces

\n

The main consideration to make once you've decided to use an email interface \n is how to deal with input and output, commands and feedback. Here's what we \n have to play with when building an email interface:

\n
    \n

  • The email address
    \n Addresses can contain information, like script_submit_123@domain.com

    • \n

  • Additional headers
    \n You can make up headers with email, but you can't \n easily edit these headers in most email programs.

    • \n

  • The body of the text.
    \n Which can be text only or HTML

    • \n

  • Attachments.
    \n eFax's\n email Fax interface (screenshot) \n for example treats any attachment as a document to be faxed by the service.

    • \n
\n

These technical constraints limit the things we can do with email. Given these \n constraints, and looking around on the web, here are the three basic styles \n of email interface interaction:

\n
    \n

  1. Command line style
    \n The subject and body of the email contain text \n commands like "subscribe" or "start-content".

    \n

    Command line style interaction requires the user to learn to use certain \n commands, and introduces possible errors: commands can be misspelled, wrong \n commands can be given. Command line style interfaces are emailed back to the \n server, allowing for asynchronous interaction. Because of their learning \n curve, command line style email interfaces are useful mainly for power users. \n A special case of command line interaction is when the user is requested to \n reply to an email that already contains the relevant commands - mailing list \n software often uses this to confirm subscriptions.

    \n
    2. \n

  2. Link based style
    \n The email contains links like "Delete this \n entry: http://domain.com/delete?12" that can execute commands.

    \n

    Link based style interaction is easy (just click the link) and error \n free, but requires the user to be online while using it, thus loosing one of \n the key benefits of email interfaces: asynchronicity. They retain the speed \n advantage, which is why these are also widely used for simple interactions like \n subscribing or unsubscribing.

    3. \n

  3. HTML based style
    \n HTML forms, posted to a webpage. Browser opens \n after posting.

    \n

    HTML based style interaction allows for a rich interaction environment, \n providing all the HTML widgets (dropdowns, checkboxes, ...) people are used \n to using. However, it requires people to be online, thus again loosing a main \n advantage of email interfaces, and it still retains all of the disadvantages \n of email interaction.

    4. \n

\n

\n Question:
\n Could an HTML form that is sent to someone be replied to, \n keeping the form intact? The form is filled in and emailed (not POSTed) \n to the server.\n

\n

Would it be possible to make the form values be transmitted \n to the server via a reply-to? This would provide the rich interaction possibilities \n of HTML and yet allow for asynchronous interaction at the same time.

\n
\n

Conventions

\n

There are a few conventions that have already evolved for email interfaces:

\n
    \n
  • Explain the user where the email comes from and what to do next if they \n aren't used to the interaction. A good example is the "This is an automatic \n email, no reply is necessary".
    • \n
  • Have subject lines that are easily filterable, preferably contained within \n square brackets. Having a subject line include something like [feedback domain.com] \n makes it easier for people to manage their incoming email.
    • \n
  • Avoid too many email being sent, rather compile them into one big email \n if possible. Nobody likes receiving heaps of email.
    • \n
\n

Conclusion

\n

Email interfaces can be powerful tools that support repetitive one step tasks, \n and offer some unique productivity advantages for people with limited time and \n Internet connectivity.

\n

The design of email interfaces has typically been left to developers with little \n time and resources to spend on this. It is high time the interaction design \n community took up the unglamorous task of designing quality email interactions.
