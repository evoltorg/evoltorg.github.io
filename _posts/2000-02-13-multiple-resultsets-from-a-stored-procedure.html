\n In Cold Fusion, and in ASP (using ADO), you can call one stored procedure, and get back multiple resultsets, as long as the database can provide them. One such database is MSSQL7.0.

Advantages to returning multiple resultsets: - Reduce network traffic slightly, in that your script only makes the ODBC / ADO / OLEDB call once instead of 6 times for 6 resultsets. - have one procedure that collects all data required for a given web page.

\n

Difference Between CF and ASP: - In CF the code is still pretty clean. You assign a query name to each of the returned resultsets and can access them out of order if you want.\n

- In ASP w/ADO, you access the first, then the second, then the 3rd, etc... and (AFAIK) there's no going back to the previous set.

Sample code for CF:

\n\n
\n<CFSTOREDPROC  \n    PROCEDURE="spMyProcName"  \n    DATASOURCE="myDataSource"  \n    USERNAME="myusername"  \n    PASSWORD="mypassword"  \n    RETURNCODE="Yes">  \n        <CFPROCPARAM  \n            TYPE="In"  \n            CFSQLTYPE="CF_SQL_VARCHAR"  \n            DBVARNAME="@myname"  \n            VALUE="#session.myname#"  \n            NULL="No">  \n        <CFPROCPARAM  \n            TYPE="In"  \n            CFSQLTYPE="CF_SQL_VARCHAR"  \n            DBVARNAME="@targetname"  \n            VALUE="#session.targetname#"  \n            NULL="No">  \n        <CFPROCPARAM  \n            TYPE="Out"  \n            CFSQLTYPE="CF_SQL_VARCHAR"  \n            DBVARNAME="@NewKey"  \n            CFVARNAME="MyNewKey"  \n            NULL="Yes">  
<!--- List of results ---> 
    <CFPROCRESULT  \n        NAME="firstresult">  \n    <CFPROCRESULT  \n        NAME="secondset"  \n        RESULTSET="2">  \n    <CFPROCRESULT  \n        NAME="thirdset"  \n        RESULTSET="3">  \n    <CFPROCRESULT  \n        NAME="fourthset"  \n        RESULTSET="4">  \n    <CFPROCRESULT  \n        NAME="fifthset"  \n        RESULTSET="5">  \n    <CFPROCRESULT  \n        NAME="sixthset"  \n        RESULTSET="6">  \n    <CFPROCRESULT  \n        NAME="seventhset"  \n        RESULTSET="7">  \n    <CFPROCRESULT  \n        NAME="eighthset"  \n        RESULTSET="8">  \n</CFSTOREDPROC>  \n
\n

\n... you use the query like this:

\n

\n \n <CFOUTPUT QUERY="firstresult"> \n ... your code here ... \n </CFOUTPUT> \n \n

ASP Sample Code:

\nFor those of you have have Visual Basic or the ADO SDK, please see the ADO help file, click 'Jump to URL' from the command menu, type:

\n

mk:@MSITStore:C:\Program%20Files\Common%20Files\System\Ado\ado210.chm::/htm/mdmthnextrecx.htm\n

\n

(or just look up the keyword 'Nextrecordset')\n

\nFor the benefit of others, here's the jist of it: (although their example shows 2 Select statements instead of one stored proc, I prefer using a stored proc)\n

\n

\nDim rstCompound as ADODB.Recordset ...\nSet rstCompound = New ADODB.Recordset rstCompound.Open "SELECT * FROM authors; " & _ "SELECT * FROM stores; " & _ "SELECT * FROM jobs", strCnn, , , adCmdText
intCount = 1 Do Until rstCompound Is Nothing \nDebug.Print "Contents of recordset #" \n& intCount Do While Not rstCompound.EOF Debug.Print , rstCompound.Fields(0), _ rstCompound.Fields(1) rstCompound.MoveNext Loop Set rstCompound = rstCompound.NextRecordset intCount = intCount + 1 Loop
\n

I highly recommend leaving as much of the database logic, and the selects, on the database in stored procedures. You'll be giving your DBA more work that's they're really good at, and freeing up the web developer to be doing more creative web stuff.
