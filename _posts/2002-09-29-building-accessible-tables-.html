CSS and \nXHTML have given tables a pretty rough ride in recent times. Of course, \n this is the fault of just about all web developers who have at some point in \n their career used them for laying out page elements. This article is not about \n using tables for layout. It is about how to use tables to display information \n in an accessible manner. Use XHTML\n and CSS all you want for layout, but at the \n end of the day (if there is one on the internet) so long as there are informational \n relationships there will be tables. In fact, to attempt to display any of complex \n information relationships without a table is a mistake. If you have information \n to display, use tables and use them well.

\n

Table Basics

\n

If I create a plan of what I intended to achieve yesterday and place it in a table, to a sighted \n user it can be a great aid:

\n \n \n \n \n \n \n \n \n \n \n \n \n \n \n \n \n \n \n \n \n \n \n \n \n \n
\n Today's action plan: \n
TimePlanned TaskPreferred TaskWhat I actually did
10 - 12Catch up on replying to my emails.Have a lie in.Surfed aimlessly.
12-1Have Lunch.Have Lunch.Had Breakfast.
1-5Finish that project. Go to video store.Caught up on my emails.
\n

Blind and other severely vision impaired users rely on assistive technologies \n to render web content. All text can be fed through a screen reader or refreshable \n braille display to be presented to the user. However, despite containing text, \n the table above could make a real mess of things without a few extra behind \n the scenes accessibility enhancements. To an assistive browser our table could \n well look like this:

\n
Time Planned Task Preferred Task What I actually did
\n 10 - 12 Catch up on replying to my emails. Have a lie in. Surfed aimlessly.
\n 12-1 Have Lunch. Have Lunch. Had Breakfast.
\n 1-5 Finish that project. Go to video store. Caught up on my emails.
\n

Ok, we can still kind of make a little sense of it. That is because we can see it. A blind \n user can't easily scan back up and check what the third piece of information \n relates to. Was it what I actually did, or what I would have preferred to do? \n This is why we need to create relationships within the code of our table. Thankfully, \n from HTML4.0 a \n bucketful of accessibility enhancements were introduced to enable \n the conscientious developer to do this.

\n

Before we see how this is done let's go back to first grade. A table is created \n with a table element. Everything in the table is contained between <table></table>. \n Beneath this we can place rows with the <tr></tr> element. Each \n row can have any number of cells using the <td></td>. Rows can alternatively \n contain the header element, <th></th>, which we will see are basically \n fancy cells with allow us to add turbo-charged accessibility to our table.

\n

Phew, you are still here.

\n

Explaining Our Table

\n

The first thing we should include in any table is a caption and a summary. \n The caption will be visible to all users, the summary is a bit of hidden code \n for special browsers. The caption comes right under the opening table tag:

\n

<table><caption>Today's action plan: </caption>...

\n

It should provide the user with a succinct and straight to the point description \n of the table's content. The caption will be rendered on the screen when your \n document is accessed so keep it simple but coherent.

\n

The summary is a place to guide non-visual agents and is an attribute taken \n by the opening table tag. Here you can describe the table in a bit more detail \n especially with regards to structure:

\n

<table summary ="This table charts my activities for the day \n based on three criteria for each of three periods of the day: What I was supposed \n to so; what I would have preferred to do and what I actually did. ">...

\n

Our table makes much more sense now to a blind user. Actually, she can probably \n figure out what it should look like. Many visually impaired users actually prefer \n to be labeled "vision impaired". Just because they can't see, it doesn't \n mean they can't visualize! Another point to keep in mind is that the summary \n attribute is not rendered on screen, so we can have the time of our life with \n it. We just need to make sure that what we write in there is relevant and provides \n assistance. Don't write all of the table data in there, just describe the relationship \n of data in the table.

\n

Relationship advice.

\n

Now we know what our table is about, we need to show the browser how information \n is related. This will help us overcome the problem of of information being rendered \n in continuous and seemingly unstructured lines. This is where we go to work \n with our table headers. The table above has 4 cells (or columns) per row, so \n we will need to add 4 headers to show what these columns represent. The best \n place for headers is in our first row, right below the <caption></caption> \n element:

\n

Here is what it looks like for the table above:

\n
<tr>\n  <th id="h1">Time.</th>\n  <th id="h2">Planned Task</th>\n  <th id="h3">Preferred Task.</th>\n  <th id="h4">What I actually did.</th>\n</tr>
\n

Notice that each <td> tag has taken an attribute called id which has \n a unique identifier. I will show you now how we use these unique identifiers \n to create relationships for the information in our table, and what the outcome \n is. We now need to link all of our columns containing the data to their appropriate \n header. Look how easy this is by looking at the first row of data from the table \n above:

\n
<tr>\n  <td headers="h1">10 - 12</td>\n  <td headers="h2">Catch up on replying to my emails.</td>\n  <td headers="h3">Have a lie in.</td>\n  <td headers="h4">Surfed aimlessly.</td>\n</tr>
\n

\n What we have done here is created a link between a piece of information and \n its header by incorporating a headers attribute into each cell that refers to \n the relevant id we created in our <th> id attribute. All we need to do \n is make sure we include the same references in every row we create in our table. \n So its not even difficult to achieve with dynamic output from a scripted page \n such as one produced with asp or\n php. But what does all that extra stuff do. \n The best way to explain is to show you how a screen reader would deal with that \n information. Here is the output for our example row as seen by a screen reader:

\n

Time: 10 -12
\n Planned task: Catch up on replying to my emails.
\n Preferred Task: Have a lie in.
\n What I actually did: Surfed aimlessly.

\n

...and so on for every row of data we have. To a sighted user, it may seem \n a drag that every piece of data is proceeded by its header. To a blind or vision\n impaired user it can be the only way to keep track of a relationships in a complex \n set of data. If your headers are made up of quite long information, this can \n be quite a nuisance, but there is also a way to deal with this. Your headers \n can also take on an abbr attribute where you type in an abbreviated reference \n to your header title. here is an example using our table's headers:

\n
<tr>\n  <th id="h1">Time.</th>\n  <th id="h2" abbr="planned">Planned Task</th>\n  <th id="h3" abbr="preferred">Preferred Task.</th>\n  <th id="h4" abbr="actual">What I actually did.</th>\n</tr>
\n

The cool thing about this is that on its first pass, the reader will render \n the full header, then for every subsequent row of data it will only read out \n the abbreviated value for the header. This gives the user a reference point \n without having to listen to the full header at every cell.

\n

Digging Deeper

\n

Before I close up here,I will mention that HTML4.0 \n also introduced an axis attribute which is still to gain support amongst many \n assistive technologies. The axis attribute is to help explain more complex informational \n structures and if you want to take a look into it you will see that it is well \n outside the scope of this article.On close inspection the axis attribute also \n looks suitable for more complex data mining as well as accessibility - once \n the technology is in place. For more information about the axis attribute read \n the w3 \n recommendations for tables. In the meantime I hope you find these simple \n techniques for making your tables accessible of use and start implementing them \n into your own sites.
